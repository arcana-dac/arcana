# RB-001.1 Email Triage & Header Analysis

## Objective
Determine whether a reported email is malicious by analysing headers, sender authenticity, and embedded indicators.

---

## Inputs

- Full email headers (raw)
- Email body (raw + rendered)
- Sender address
- Attachments (if any)
- Embedded URLs
- Email security alert (if triggered)

---

## Tools

- Email platform:
  - Microsoft 365 (Message Trace, Defender)
  - Google Workspace (Email Log Search)
- Header analysis tools:
  - Built-in analyzers / MXToolbox
- Threat intel:
  - VirusTotal
  - URLScan
- Internal logging:
  - SIEM / log platform

---

## Steps

### 1. Collect Full Email Data

- Retrieve full email headers (not partial)
- Capture:
  - Sender (From, Reply-To)
  - Subject
  - Timestamp
  - Message-ID

---

### 2. Analyse Authentication Results

Check:

- SPF → pass/fail
- DKIM → pass/fail
- DMARC → pass/fail/alignment

**Indicators:**
- SPF fail or softfail
- DKIM missing or invalid
- DMARC fail

---

### 3. Validate Sender Identity

- Compare:
  - Display name vs actual email
  - Domain similarity (typosquatting, homoglyphs)
- Check:
  - Newly registered domains
  - External sender spoofing internal user

---

### 4. Review Email Routing

Analyse `Received:` headers:

- Originating IP
- Sending infrastructure
- Unusual relays or geolocation anomalies

---

### 5. Extract Indicators

Pull out:

- All URLs (including obfuscated)
- Attachment names + hashes
- Reply-To domain (often different)

---

### 6. Perform Initial Reputation Checks

- URLs → VirusTotal / URLScan
- Domains → passive DNS / reputation
- File hashes → malware databases

---

### 7. Assess Email Content

Look for:

- Urgency / pressure tactics
- Credential prompts
- Financial requests
- Suspicious attachments
- Link text vs actual destination mismatch

---

## Decision Points

| Condition | Action |
|----------|--------|
| Clearly benign | Close incident, document findings |
| Suspicious but unconfirmed | Escalate to URL/attachment analysis |
| Malicious indicators present | Proceed with PB-001 Step 2 |

---

## Output

- Verdict: `benign / suspicious / malicious`
- Extracted IOCs:
  - Sender domain/IP
  - URLs
  - File hashes
- Initial scope (single email vs campaign)

---

## Automation Hooks

- Auto-extract URLs and attachments
- Auto-run header parsing
- Auto-enrich with threat intel
- Auto-create incident ticket

---

## Notes

- Display name spoofing is common—never trust it alone
- SPF/DKIM pass does NOT guarantee legitimacy
- Always check Reply-To vs From mismatch
